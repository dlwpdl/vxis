Description상세 설명
WHAT — The /rest/products/search endpoint passes the user-controlled q parameter into a backend SQL query without sufficient parameterization or input validation. The observed error-based SQL injection condition indicates that attacker-supplied payloads can alter the query syntax and force database error messages to be returned to the client, confirming that the search function is vulnerable to server-side SQL injection. In a public-facing application like OWASP Juice Shop, this typically provides a direct path from a simple search field to unauthorized data extraction, credential compromise, and application-wide abuse.
HOW — Step 1: Send a benign request such as GET /rest/products/search?q=apple and note the normal product results. Step 2: Inject SQL metacharacters or a syntactically invalid UNION payload such as ' UNION SELECT 1,2,3-- to deliberately break the query. Step 3: Observe an error response, stack trace, or database exception that changes based on the input, confirming the query is being interpreted by the database engine. Step 4: Refine the payload using error-based techniques to extract schema details, table names, and column counts. Step 5: Enumerate sensitive tables such as users, baskets, orders, and auth-related data, then pivot to account compromise or session theft.
IMPACT —
- Exposure of sensitive application data stored in the database, potentially including user accounts, email addresses, password hashes, order history, and address/PII records; this can trigger GDPR/UK GDPR, CCPA, or local breach notification obligations depending on user residency.
- Unauthorized access to authentication material or account metadata can enable credential stuffing, password cracking, account takeover, and impersonation of customers or administrators.
- Database integrity can be affected if the injection is later extended to write operations, allowing product tampering, fraud, or defacement that directly impacts revenue and customer trust.

Identifiers식별자

CWE: CWE-915, CWE-20, CWE-639

Description상세 설명
WHAT — The /api/users/ registration endpoint appears to accept attacker-supplied fields beyond the intended public account attributes, indicating a mass assignment / over-posting flaw. In this condition, a client can submit JSON such as email, password, and role in a single request and have sensitive server-side attributes persisted without explicit server-side allowlisting. In practical terms, an unauthenticated or low-privileged actor may be able to create accounts with elevated roles, bypass intended onboarding workflows, or modify fields that should only be set by privileged services.
HOW — Step 1: Send a registration request to the public user creation endpoint with the normal required fields (for example, email and password). Step 2: Add one or more unexpected properties such as role, isAdmin, tenantId, or disabled to the same JSON body. Step 3: Observe whether the API accepts and stores these fields rather than rejecting or ignoring them. Step 4: Authenticate with the newly created account and verify whether the injected role or privilege flag is honored by downstream authorization checks. Step 5: Repeat with alternate privileged attributes to determine whether the issue affects only initial account creation or also profile update flows.
IMPACT —
- An attacker may self-register or modify an account into an elevated role, resulting in unauthorized access to administrative functions, user management, order manipulation, or content moderation features.
- Sensitive data stored behind role-based access controls may become accessible, including customer profiles, order history, payment-related metadata, and internal administrative views, increasing the likelihood of reportable data exposure under privacy regulations such as GDPR or CCPA.
- If the same object mapping pattern is reused in account updates, the flaw can enable persistent privilege escalation or account takeover for existing users, expanding impact from a single account to the entire tenant or application population.

Step 4: Assess application behavior across login, account, and product flows to understand where injected content, browser extensions, or compromised network intermediaries could leverage the missing protections. Step 5: Use this as an enabling condition for higher-impact attacks such as XSS exploitation, session theft, and content tampering.
IMPACT —
