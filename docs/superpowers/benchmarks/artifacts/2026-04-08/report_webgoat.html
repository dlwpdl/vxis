Identifiers식별자

Description상세 설명
WHAT — The /WebGoat/start.mvc endpoint appears to trust a client-supplied id parameter when loading lesson content or state, and does not sufficiently enforce object-level authorization before returning the requested resource. In practice, this means an authenticated or unauthenticated caller can supply a different id value and access data or lesson state that should be restricted to another user, lesson context, or privilege level. This is a classic Insecure Direct Object Reference (IDOR) / Broken Object Level Authorization (BOLA) condition, where the application exposes a direct object reference and relies on obscurity rather than server-side ownership checks.
HOW — Step 1: Navigate to the application and observe that the login page is reachable at /WebGoat/login, with the vulnerable flow ultimately invoking /WebGoat/start.mvc. Step 2: Intercept a legitimate request that includes id=1 and note the response content tied to that object identifier. Step 3: Modify the id parameter to another value (for example, 2, 3, or a higher sequential number) while keeping the same session or even no session, depending on access control behavior. Step 4: Compare responses and observe that different object data or lesson content is returned without verifying that the caller is entitled to access that record. Step 5: Enumerate ids systematically to identify additional accessible objects, user-specific state, or privileged lesson content.
IMPACT —
- Unauthorized disclosure of application data, lesson state, or other object-bound records associated with different users or roles, which can include training progress, answers, identifiers, or administrative content depending on deployment configuration.
- If the vulnerable object maps to real tenant, user, or course records in a non-training deployment, the issue can expose personal data or internal business content, creating confidentiality loss and possible privacy/regulatory exposure (for example, GDPR/CCPA obligations if user identifiers or profile data are revealed).
- IDOR primitives commonly enable privilege escalation and lateral abuse by chaining into account enumeration, unauthorized modification, or harvesting of trusted identifiers for phishing, fraud, or further access control bypasses.

Recommendation권고 사항
Immediate: Enforce server-side object ownership checks on /WebGoat/start.mvc before any object is loaded; deny access with 403 when the requested id is not owned by or authorized for the current user/session.

HIGH 9.1 CVSS:3.1/AV:N/AC:L/… VXIS-002 / broken_authentication Open
Identifiers식별자

CWE: CWE-347, CWE-287, CWE-290

Description상세 설명
WHAT — The application accepts a malformed or attacker-modified JWT on the / route and still treats the token as valid, allowing authentication to be bypassed without possessing a legitimately issued token. The observed payload uses an RS256 header with a trivially crafted body (sub=test) and the application accepted the manipulated token, indicating broken JWT verification and/or trust in untrusted claims. In practice, this means the login boundary can be bypassed and downstream access control decisions may be made on attacker-supplied identity data rather than a server-validated session.
HOW — Step 1: Intercept the authentication flow to identify where the application stores or forwards JWTs. Step 2: Modify the JWT header/payload, keeping the signature placeholder or altering claims such as sub, role, or user_id. Step 3: Submit the tampered token to the protected / endpoint or any JWT-backed request. Step 4: Observe that the server accepts the token and grants access, demonstrating that signature validation, issuer/audience validation, or algorithm enforcement is missing or incorrectly implemented. Step 5: Use the accepted identity to access authenticated functionality and enumerate other user-facing or administrative endpoints.
IMPACT —
- Authentication bypass can expose all functionality reachable after login, including personal profile data, training content, administrative functions, or account settings, depending on how authorization is implemented.
- If the JWT carries role, username, or tenant claims, attacker-controlled values may be accepted as authoritative, enabling privilege escalation, tenant hopping, or unauthorized data access across user accounts.
- In a production deployment, this issue can trigger unauthorized access to regulated data such as personal identifiers, contact information, learning records, or audit history, creating breach-notification and compliance exposure under GDPR, CCPA, or similar privacy regimes.
