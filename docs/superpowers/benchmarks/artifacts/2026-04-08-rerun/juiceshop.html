Identifiers식별자

CWE: CWE-693, CWE-16

Description상세 설명
WHAT — The application is missing several baseline security response headers on the public web root, including Strict-Transport-Security (HSTS), Content-Security-Policy (CSP), X-XSS-Protection, Referrer-Policy, and Permissions-Policy. These headers do not fix an underlying logic flaw by themselves, but their absence weakens browser-side defense-in-depth and increases the blast radius of any future injection, clickjacking, data leakage, or mixed-content issue. In particular, the lack of HSTS leaves users exposed to SSL stripping and downgrade attacks when they first reach the site over HTTP, the absence of CSP materially increases the impact of any reflected/stored XSS in the application, missing Referrer-Policy can leak sensitive path/query data to third-party origins, and the absence of Permissions-Policy leaves unnecessary browser capabilities available to all page contexts.
HOW — Step 1: Request the application root over HTTP and inspect the response headers. Step 2: Confirm that the response does not include Strict-Transport-Security, Content-Security-Policy, X-XSS-Protection, Referrer-Policy, or Permissions-Policy. Step 3: Load the application in a browser and observe that it renders normally, demonstrating that the issue is not availability-related but a configuration gap. Step 4: Chain the missing headers with an application bug such as XSS, open redirect, or third-party script compromise to increase the likelihood of successful exploitation and reduce detection opportunities. Step 5: If users can initially reach the site over plain HTTP or are redirected from bookmarks, an active network attacker can strip or manipulate the first request before HTTPS enforcement occurs because HSTS is not present.
IMPACT —
- Increases the severity of any client-side exploit across the platform: a single XSS, malicious widget injection, or compromised dependency is more likely to lead to session theft, account takeover, and access to user data when CSP is absent.
- Exposes users to downgrade and SSL-stripping attacks on first visit or on non-HSTS paths, which can result in credential capture, session hijacking, and disclosure of authentication cookies or personal data.
- May leak sensitive URL parameters, reset links, or internal path information via the Referrer header to third-party domains, analytics providers, or embedded content, creating privacy and compliance risk (for example GDPR/CCPA exposure if personal data is included in URLs).
- Allows unnecessary browser features such as camera, microphone, geolocation, clipboard, and fullscreen to remain available unless explicitly blocked, expanding the impact of any script execution or malicious third-party content.

Identifiers식별자

Description상세 설명
WHAT — The application responds with a permissive Cross-Origin Resource Sharing (CORS) policy on the root path, advertising Access-Control-Allow-Origin: * for a publicly reachable web application that appears to host authenticated user functionality. In practice, this means any arbitrary origin can read responses from the site when the browser permits the request flow, which is especially dangerous if the application ever returns sensitive data on credentialed or authenticated endpoints without strict origin validation. While the observed issue is at the application edge rather than a single business endpoint, it weakens the browser’s same-origin protection and can enable cross-site data theft, session riding, and token exfiltration when combined with other weaknesses such as weak authentication handling, reflected secrets, or permissive credential settings.
HOW — Step 1: Host attacker-controlled HTML/JavaScript on an external origin such as https://evil.example. Step 2: From that origin, issue a cross-origin fetch or XMLHttpRequest to http://localhost:3000/ with the Origin header set to the attacker domain. Step 3: Observe that the server replies with Access-Control-Allow-Origin: * (and, if present on other routes, may also allow credentials or broadly expose headers). Step 4: Expand testing to authenticated API routes and application resources to determine whether the same policy applies beyond /, allowing the attacker origin to read product, account, or session-related responses. Step 5: If any sensitive endpoint is exposed under the same policy, use browser-based JavaScript to exfiltrate the response body to the attacker server.
IMPACT —
- Cross-origin read access may expose authenticated user data, profile details, orders, baskets, or CSRF tokens if any such content is returned by endpoints covered by the same policy, creating confidentiality impact for all users whose sessions can be reached from a browser.
- If credentialed requests are permitted elsewhere in the application, an attacker can weaponize a victim’s browser to retrieve sensitive data without direct server-side authentication bypass, increasing the blast radius of any future XSS, phishing, or malicious link delivery.
- From a regulatory perspective, the unauthorized disclosure of personal data, purchase history, or account identifiers may trigger privacy obligations such as GDPR/UK GDPR incident assessment and internal breach handling depending on the tenant data exposed.

Description상세 설명
WHAT — The application’s root page (`/`) is missing baseline browser security headers, specifically `Content-Security-Policy` and `Strict-Transport-Security`. CSP is a primary client-side containment control that limits script execution, resource loading, and the blast radius of XSS and data exfiltration. HSTS instructs browsers to enforce HTTPS-only access after the first secure visit, reducing the risk of SSL stripping and session interception on hostile networks. While this issue is not directly exploitable on its own, it weakens the overall security posture of the application and materially increases the impact of otherwise moderate client-side defects.
HOW — Step 1: Request the application root and inspect the response headers. Step 2: Confirm that `content-security-policy` and `strict-transport-security` are absent from the response. Step 3: Compare behavior with a hardened deployment to validate that the application does not enforce a restrictive script/resource policy or HTTPS upgrade policy. Step 4: Assess likely attack chaining opportunities: any reflected/stored XSS, DOM XSS, third-party script compromise, or hostile network position becomes more dangerous because the browser has no CSP/HSTS backstop. Step 5: Use the missing headers as a control-gap indicator to prioritize remediation before enabling features that increase client-side attack surface.
IMPACT —
- Increases the likelihood and severity of cross-site scripting impact by allowing injected scripts to execute without CSP-based containment, which can expose authentication tokens, CSRF tokens, personal data, and shopping/cart state.
- Leaves users more vulnerable to man-in-the-middle downgrade attacks on first connection or non-HSTS-capable flows, which can expose session cookies, credentials, and account actions on untrusted networks.
