Identifiers식별자

CWE: CWE-693, CWE-16

Description상세 설명
WHAT — The WebGoat login page at http://localhost:8080/WebGoat does not emit several baseline browser security headers, including Strict-Transport-Security, Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, X-XSS-Protection, Referrer-Policy, and Permissions-Policy. While this is not a direct code-execution flaw by itself, the absence of these controls materially weakens the browser-side security posture of the authentication surface, increasing the likelihood that a separate injection, phishing, clickjacking, mixed-content, or data-exfiltration issue can be converted into a full compromise.
HOW — Step 1: Load the login page and inspect the HTTP response headers. Step 2: Confirm that the response lacks HSTS, CSP, frame-busting, MIME-sniffing protection, referrer minimization, and browser feature restrictions. Step 3: Place the application behind a reverse proxy, shared hosting stack, or content injection point and observe that any downstream mistake is now less constrained by browser-enforced defenses. Step 4: Use the missing X-Frame-Options / frame-ancestors protection to embed the login page in an attacker-controlled frame and attempt UI redress or clickjacking if any sensitive action is exposed post-login. Step 5: Use the missing CSP and related headers to increase the impact of any future reflected or stored XSS by permitting script execution paths, data exfiltration, and unauthorized network requests. Step 6: Because HSTS is absent, if the application is ever served over HTTPS or accessed through a secure front end, an active network attacker can attempt SSL-stripping or downgrade opportunities on first visit until a secure-cookie policy is enforced.
IMPACT —
- Increases the likelihood that a credential-harvesting or session hijacking attack against the login flow will succeed, affecting all users who authenticate through this page, including administrators and privileged operators.
- Weakens defenses required to contain browser-based attacks, which can expose usernames, passwords, session cookies, CSRF tokens, and application data if another vulnerability is present; this can trigger incident response, account reset, and potentially privacy obligations depending on the data accessed.
- Fails common hardening expectations used in baseline security benchmarks and many compliance programs (for example, internal secure configuration standards, CIS-style hardening, and browser-side protections expected for regulated environments), increasing audit findings and remediation cost.

Recommendation권고 사항
Immediate: Add the missing security headers at the web server or application gateway layer for all responses, especially the login page. Enforce HTTPS and set Strict-Transport-Security only after confirming the entire site is TLS-ready.
Short-term: Implement a restrictive Content-Security-Policy, add X-Frame-Options or frame-ancestors, set X-Content-Type-Options: nosniff, define a minimal Referrer-Policy, and apply a Permissions-Policy that disables unused browser features. Validate the headers through automated tests.
Long-term: Standardize a secure HTTP response header baseline across all services, manage it centrally in the reverse proxy or framework middleware, and add security-header checks to CI/CD and release gates so regressions are detected before deployment.즉시 조치: 웹 서버 또는 애플리케이션 게이트웨이 계층에서 모든 응답, 특히 로그인 페이지에 누락된 보안 헤더를 추가합니다. 전체 사이트가 TLS 사용에 준비되었는지 확인한 후에만 HTTPS 강제 및 Strict-Transport-Security 적용을 활성화합니다.
단기 조치: 제한적인 Content-Security-Policy를 도입하고, X-Frame-Options 또는 frame-ancestors를 설정하며, X-Content-Type-Options: nosniff, 최소 권한의 Referrer-Policy, 불필요한 브라우저 기능을 비활성화하는 Permissions-Policy를 적용합니다. 헤더가 자동 테스트로 검증되도록 구성합니다.
장기 조치: 모든 서비스에 대해 안전한 HTTP 응답 헤더 기준선을 표준화하고, 리버스 프록시 또는 프레임워크 미들웨어에서 중앙 관리하며, CI/CD와 릴리스 게이트에 보안 헤더 검사를 추가하여 배포 전 회귀를 탐지하도록 합니다.

Identifiers식별자

CWE: CWE-209

Description상세 설명
WHAT — The /login endpoint returns a server-side error page that discloses Java stack trace details, including source file names and line numbers (for example, patterns matching `at com.example.Class.method(Class.java:123)`) when a 404 or other request handling error is triggered. This is an information disclosure condition because the application reveals internal implementation details that should be suppressed in production. In a login workflow, this commonly exposes framework versions, package names, controller class names, and code paths that materially reduce the effort required to identify additional vulnerabilities.
HOW — Step 1: Browse to the login page at `http://localhost:8080/WebGoat/login` and confirm the normal form behavior. Step 2: Trigger an error condition by requesting a malformed or non-existent login-related path, such as `/login/..`, `/login/does-not-exist`, or by submitting an unexpected HTTP method or parameter shape that causes the backend to route to an error handler. Step 3: Observe that the server returns a 404 error page containing debug details rather than a generic production-safe response. Step 4: Extract the stack trace fragments, source file names, and line numbers from the response body and use them to map the backend technology stack and codebase structure. Step 5: Use the leaked implementation details to accelerate enumeration of adjacent endpoints, parameter names, and error-handling behaviors for further exploitation.
IMPACT —
- Discloses internal source file names, class names, and line numbers, which lowers the cost of targeted exploitation and accelerates discovery of additional flaws such as authentication bypass, input validation errors, or unsafe deserialization.
- Reveals framework and deployment details that can assist an attacker in crafting precise payloads, fingerprinting vulnerable library versions, and identifying compensating controls that are absent or misconfigured.
- Increases risk of chained compromise of user credentials and session data by helping an attacker locate the exact login implementation, debug endpoints, and exception paths that may expose sensitive request context.
- From a compliance perspective, error pages that reveal internal application state can undermine secure coding and production hardening requirements under common security baselines (for example, PCI DSS logging and error-handling expectations, and general privacy/security obligations where implementation details can facilitate unauthorized access).
